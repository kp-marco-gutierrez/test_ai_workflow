The shift

You describe what.
The pipeline figures out how.

The human owns

  • One capability, written as a BDD spec — Given / When / Then
  • Small and concrete enough to build and test as a single unit
  • The decision to start: a @go-develop comment

The pipeline owns

  • Writing the tests, then the implementation
  • Running every test, reviewing the code, fixing the feedback
  • Resolving conflicts, merging, and deploying

Engineers spend their attention on the intent, not the boilerplate.

Steps 1–2 · Intake
01

A spec is the contract.

File itOpen a GitHub issue from the BDD Spec template — a Gherkin Feature with Given / When / Then.
TriggerComment @go-develop when it's ready. That one comment runs the whole pipeline.
GuardrailA deterministic check rejects vague specs and ones too big to test as a unit — and suggests how to split them.

Everything downstream is derived from this spec, so it has to be small, concrete, and checkable.

Steps 3–4 · Tests first
02

Write the tests. Watch them fail.

GenerateClaude turns the spec into executable tests and opens a draft PR — no implementation yet.
Test QAChatGPT (the red team) checks the tests faithfully cover the spec.
✗ Dry-run gate — the tests must fail against the empty codebase

A failing dry-run is the proof the tests are real — that they can't silently auto-pass later.

Steps 5–6 · Build to green
03

Code until the tests pass.

ImplementClaude writes the minimal code to satisfy the failing tests, fed the real CI output each round.
AuthorityClaude never decides if it passed — it only generates. The CI runs the tests.
✓ Verify gate — the tests must pass, then the PR leaves draft

Red, then green — proven by the machine, not claimed by the author.

Steps 7–8 · Review
04

Adversarial review, then approval.

ReviewChatGPT reviews the implementation and blocks only on real issues — correctness, security, spec. Polish is a suggestion.
AddressClaude fixes the blocking points; the cycle repeats until the reviewer approves.
BackstopIf it can't converge, it's capped and handed to a human — it never loops forever, and never merges unreviewed.

Two different models: one builds, one critiques. Neither grades its own homework.

Step 9 · Ship
05

It merges and deploys itself.

ApproveA formal approval is posted, the full CI suite runs one more time.
MergeSquash-merge to main; the originating issue closes automatically.
DeployGitHub Pages redeploys the result. When main moves, open PRs auto-resync and re-verify.

From @go-develop to a live deploy — no human hands on the keyboard in between.

Why you can trust it

AI generates.
Deterministic CI judges.

The split is the whole point. The AI only ever produces artifacts — tests, code, fixes, reviews. Whether the tests pass or fail is decided by a plain test run, every time. The machine is never the judge of its own work, so "it looks done" can never stand in for "it is green."

AI does

  • Generate tests from the spec
  • Generate code to pass them
  • Generate reviews & fixes

CI decides

  • Run the tests — must fail, then must pass
  • Run the full suite before merge
  • Resolve conflicts toward main

Safety rails

Autonomous, not unsupervised.

No auto-passTests must fail before they're allowed to pass — bogus tests can't sneak through.
Right-sizedThe guardrail refuses specs too big to test as one unit and proposes a split.
Real issuesReview blocks on correctness, security and spec — not taste — so it converges.
BoundedEvery loop is capped; a stuck change is escalated to a human, never merged blind.

Humans step in at exactly two moments: writing the spec, and resolving a genuine stall.
